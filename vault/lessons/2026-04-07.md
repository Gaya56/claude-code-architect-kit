# Lessons Learned — 2026-04-07

Two major builds today: a complete `.claude/` setup for [[cisco-inventory]], then an Obsidian vault and tooling upgrade for the architect-kit itself.

## Phase 1: Cisco Inventory Build

Built and pushed a full `.claude/` setup (3 agents, 3 skills, 2 hooks, 2 rules, Context7 MCP) to the `quick-macros` branch of [Gaya56/Cisco-Inventory](https://github.com/Gaya56/Cisco-Inventory/tree/quick-macros).

### Hook Gotchas

> [!warning] Hook JSON field names matter
> Use `decision` and `reason`, not `permissionDecision` and `permissionDecisionReason`. Wrong field names = hook silently ignored. See [[hook-patterns]] for the proven template.

> [!warning] Exit codes
> Use `exit 0` with JSON `{"decision":"block"}` to block. Don't rely on `exit 2` — let the JSON be the signal.

> [!warning] Guard for jq
> If `jq` isn't installed, `jq -r ... 2>/dev/null || true` silently passes. Add a `command -v jq` guard at the top of every hook.

### Permission Patterns

> [!danger] Overly broad jq permissions
> `Bash(* | jq *)` matches ANY command piped to jq — including `rm -rf / | jq .`. Tighten to `Bash(curl * | jq *)`.

### Environment Variables

> [!important] Name consistency
> `.env` had `WEBEX_TOKEN` but everything else referenced `WEBEX_API_TOKEN`. Nothing worked until fixed. Always grep the full codebase for the variable name before pushing.

### Agent Teams

- Comma-separated tools format (`tools: Bash, Read, Grep`) works fine — matches architect-kit pattern
- Lead agent spawns subagents via task description, not by name
- Skills auto-load into subagent context — no special wiring needed
- `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` required in settings.json env

### Review Process

> [!tip] Always review with agent team
> Code-reviewer agent found 3 critical bugs (hook JSON fields, missing Macro.Get permission, researcher missing WebFetch tool) that would have caused ==silent failures==. Never skip the review step. See [[review-checklist]].

### Cross-Check Pattern

> [!example] Reusable pattern
> The `/cross-check` skill (docs vs API reality) compares what official docs say against what the API actually returns. Flags gaps before you execute. Adapt this for any project with external APIs.

### Gitignore

Always create `.gitignore` with `.env` and `.claude/settings.local.json` before any push. Add it to the [[review-checklist]].

### Obsidian Skill Selection

> [!tip] Only install what you need
> Evaluated 5 official Obsidian skills. Only `obsidian-markdown` was useful for our Codespace workflow. `obsidian-cli` requires desktop Obsidian running. `obsidian-bases` is overkill until we have 50+ notes. `defuddle` duplicates WebFetch/Tavily. See [[skill-patterns]].

### Vault Structure

> [!note] Keep it flat
> The vault uses a flat structure: `projects/`, `lessons/`, `proven-patterns/`, `templates/`. No deep nesting, no over-linking. Add a file, link it from [[dashboard]], done.

## What to Do Differently Next Time

- Run the [[review-checklist]] ==before== the first push, not after
- Create `.gitignore` as the very first file in any new project directory
- Test hooks with `echo '{}' | bash hook.sh` before wiring them in settings.json
- Check `.env` variable names match across all files with `grep -r VARIABLE_NAME`
