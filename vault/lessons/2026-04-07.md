# Lessons Learned — 2026-04-07

Built a complete `.claude/` setup for [[cisco-inventory]] from scratch.

## Hook Gotchas

> [!warning] Hook JSON field names matter
> Use `decision` and `reason`, not `permissionDecision` and `permissionDecisionReason`. Wrong field names = hook silently ignored.

> [!warning] Exit codes
> Use `exit 0` with JSON `{"decision":"block"}` to block. Don't rely on `exit 2` — let the JSON be the signal.

> [!warning] Guard for jq
> If `jq` isn't installed, `jq -r ... 2>/dev/null || true` silently passes. Add a `command -v jq` guard at the top.

## Permission Patterns

> [!danger] Overly broad jq permissions
> `Bash(* | jq *)` matches ANY command piped to jq — including `rm -rf / | jq .`. Tighten to `Bash(curl * | jq *)`.

## Environment Variables

> [!important] Name consistency
> `.env` had `WEBEX_TOKEN` but everything else referenced `WEBEX_API_TOKEN`. Nothing worked until fixed. Always grep the full codebase for the variable name.

## Agent Teams

- Comma-separated tools format (`tools: Bash, Read, Grep`) works fine — matches architect-kit pattern
- Lead agent spawns subagents via task description, not by name
- Skills auto-load into subagent context — no special wiring needed
- `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` required in settings.json env

## Review Process

> [!tip] Always review with agent team
> Code-reviewer agent found 3 critical bugs (hook JSON fields, missing Macro.Get permission, researcher missing WebFetch tool) that would have caused silent failures. Never skip the review step.

## Cross-Check Pattern

The `/cross-check` skill (docs vs API reality) is a reusable pattern for any project with external APIs. Adapt it.

## Gitignore

Always create `.gitignore` with `.env` and `.claude/settings.local.json` before any push. Add it to the review checklist.
